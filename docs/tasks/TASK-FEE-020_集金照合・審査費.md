# TASK-FEE-020 集金照合・審査費

## Origin

- `docs/ETC/道場システム_ヒアリング対応項目_2026-07-21.md`
- 集金後、現金・PayPayを合わせていくらなら正しいか分からない。
- 審査件数が多く、審査費の受付が必要。

## Reality

- `payment_teacher.html`には受付済みの現金、PayPay、合計が既にある。
- PayPayエビデンスには `CONFIRMED` / `POSTED` の確認フローがある。
- `05_請求明細`は請求種別、表示名、数量、単価を持つ汎用構造である。

## Scope

1. 受付登録額を予定値として、実際に確認した現金・PayPayとの差額を表示する。
2. `03_料金マスタ`の `会費タイプ=審査費` を追加請求として作成可能にする。
3. 同月・同会員・同プランの審査費を重複作成しない。
4. 作成後は既存の会費状態Viewと現金・PayPay受付を利用する。

## Acceptance

- 現金・PayPayそれぞれの差額と合計差額が表示される。
- 一致時は一致が明示される。
- 審査費プランのQRから請求明細を作成できる。
- 同じ審査費を再読取しても請求が重複しない。

## Teacher payment status

- `payment_status.html`を先生用の会費状況画面とする。
- 指定期間の`06_入金ログ`を現金、PayPay、合計で集計する。
- 対象月の請求、入金、未払い、稽古課金回数を請求グループ単位で一覧表示する。
- 未払いのある行を先に表示する。
- 会費受付画面の未送信リスト集計と、登録済み入金の期間集計を区別する。

## Billing block reception

- 会費受付では道場、先生、課金枠が揃ってから会員受付を行う。
- `09_決済エビデンス`から`06_入金ログ`まで、道場・課金枠・先生・受付セッションを引き継ぐ。
- 本日かつ現在の道場・課金枠に登録済みの現金、PayPay、合計、件数を再取得できる。
- 初月の運用確認は`STORY-902 一か月試験運用`に従う。

## Operation prerequisite

`03_料金マスタ`へ、級段位別に次のような行を登録する。

- `plan_id`: 例 `EXAM-KYU2`
- `会費タイプ`: `審査費`
- `表示名`: 例 `2級審査費`
- `回数単価`: 正式な審査費
- `状態`: `有効`

金額と対象級段位は先生への確認後に登録する。
