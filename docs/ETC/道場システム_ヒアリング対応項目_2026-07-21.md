# 道場システム ヒアリング対応項目

- 記録日: 2026-07-21
- 対象: 運用開始に向けた機能追加
- 基本方針: Framework対応は一旦保留し、現行コード・現行設計・実際の道場運用をRealityとして進める。

## 1. ヒアリング・立会いで確認した事実

### 1.1 先生ヒアリング

1. PayPay決済は現在のところ順調に運用できている。
2. 先生が出席情報から知りたいのは、金銭情報よりも当日の参加メンバー構成である。
3. 参加メンバーに応じて、次のように稽古内容を組み立てたい。
   - 白帯が多い場合: 基本技を多くする。
   - 審査が近い会員がいる場合: 対象級・段に合った技を選ぶ。
   - 黒帯・上級者が多い場合: 技のバリエーションや質の向上を扱う。
4. 審査件数が多いため、審査費の取扱いを追加する必要がある。

### 1.2 副道場長の会費受領立会い

1. 個々の会員から集金する操作自体には、大きな問題はない。
2. 現金集金担当とは別に、PayPay入金を確認する担当が必要になりそうである。
3. 集金後、現金とPayPayを合わせて「いくらなら正しいのか」が分からず、再計算が複雑になる。

### 1.3 現行の出席管理表

1. 合気会共通様式の管理表である。
2. 審査時に合気会へ提出し、審査合格後は会員の手元に残らない。
3. 現状の出席表には現在の級・段が明示されておらず、先生は記憶または事前調査で補っている。
4. 最終的には、デジタル出席情報から共通様式を生成し、先生の確認印を得て提出する運用を目指す。

## 2. 運用開始時に創出するメリット

### 2.1 先生

- 当日の出席者の級・段と審査までの進捗を一覧で確認できる。
- 会員情報を記憶や別資料から補わず、当日の稽古内容を組み立てられる。
- 審査が近い会員を見落としにくくなる。
- 会費の予定額・実績額・未収を確認できる。

### 2.2 会員

- 自分の級・段、稽古数、審査までの残り回数を確認できる。
- 今月の出席回数や直近の出席日を確認できる。
- 紙の管理表を提出した後も、自分の稽古履歴がデジタルで残る。
- 会費・審査費の支払状況を確認できる。

### 2.3 集金・PayPay担当

- 現金とPayPayを分担して処理できる。
- 集金予定額と受領実績を照合できる。
- 未確認・未収の会員を特定できる。

## 3. 対応項目

### 3.1 出席管理系: 初期実装

| ID | 対応項目 | 目的 | 状態 |
|---|---|---|---|
| ATT-01 | 会員情報に現在の級・段を追加 | 指導判断と審査進捗計算の基礎情報にする | 実装対象 |
| ATT-02 | 出席時に登録済みの級・段をデフォルト表示 | 会員の入力負担を抑える | 実装対象 |
| ATT-03 | 級・段が違う場合は会員本人が変更 | 初期データを先生へ一括登録させず、日常運用で整備する | 実装対象 |
| ATT-04 | 変更した級・段を会員情報へ反映 | 次回以降のデフォルト値にする | 実装対象 |
| ATT-05 | 級・段の登録元・更新日を識別可能にする | 将来の先生確認方式へ移行できるようにする | 設計確認 |
| ATT-06 | 本日の出席一覧に現在の級・段を表示 | 先生が参加者構成を判断できるようにする | 実装対象 |
| ATT-07 | 一覧を級・段、審査までの残り稽古数等でソート | 稽古プランの対象者を探しやすくする | 実装対象 |
| ATT-08 | 審査が近い会員を色分け | 審査対象者の見落としを防ぐ | 実装対象 |
| ATT-09 | 会員本人へ出席・審査進捗サマリーを表示 | 会員側にもデジタル化のメリットを返す | 実装対象 |
| ATT-10 | 導入前の紙管理表の稽古数を繰越登録 | 運用開始時点から紙とデジタルの進捗を合わせる | 運用・設計確認 |

会員向けサマリーの初期表示候補:

- 現在の級・段
- 現在の審査対象稽古数 / 審査可能稽古数
- 審査可能までの残り稽古数
- 今月の出席回数
- 最終出席日
- 直近の出席日

初期段階では、紙の表形式を完全再現せず、スマートフォンで確認しやすいサマリーを優先する。

### 3.2 会費管理系: 既存機能の整理

| ID | 対応項目 | 目的 | 状態 |
|---|---|---|---|
| FEE-01 | 既存機能から集金予定総額を明示 | 集金後の正解額を示す | 現行確認後に実装 |
| FEE-02 | 現金受領合計を表示 | 現金担当の締め確認を容易にする | 現行確認後に実装 |
| FEE-03 | PayPay受領・確認合計を表示 | PayPay担当の確認範囲を明確にする | 現行確認後に実装 |
| FEE-04 | 受領合計、未収額、未払い者を表示 | 再計算の混乱をなくす | 現行確認後に実装 |
| FEE-05 | PayPay未確認・確認済みを識別 | 担当分離後も処理状況を共有する | 現行確認後に判断 |
| FEE-06 | 審査費を既存の料金・支払い構造へ追加 | 審査件数増加へ対応する | 実装対象 |
| FEE-07 | 会員本人へ会費・審査費の支払状況を表示 | 払い忘れ・二重払い・確認漏れを防ぐ | 段階導入 |

会費系は新規の請求管理を先に設計せず、現行コードに存在する機能とデータを確認し、表示・集計・DTOを整理してメリットを創出する。

### 3.3 DTO・ビュー改善

| ID | 対応項目 | 方針 |
|---|---|---|
| DTO-01 | 今回変更するサマリービューDTOを整理 | 取得元テーブルが追える番号・命名にする |
| DTO-02 | DTO項目番号をテーブル番号と対応させる | 複数テーブルからの寄せ集めを読みやすくする |
| DTO-03 | 改善範囲を直接の利用箇所に限定 | Framework化や全体リファクタリングへ広げない |

## 4. 将来対応

### 4.1 公式管理表の自動生成

1. デジタル出席記録を正本とする。
2. 審査申請時に合気会共通様式の管理表をPDF等で生成する。
3. 先生が内容を確認する。
4. 印刷後に先生の確認印を押す。
5. 合気会へ提出する。
6. 提出時点の帳票をデジタル履歴として保存する。
7. 合格後、新しい級・段の審査対象期間を開始する。

恒常的な紙・デジタルの二重運用は避ける。移行時のみ、集計結果を検証する目的で一時的に併用する。

### 4.2 将来の級・段登録

- 初期運用: 会員本人による申告・補正。
- 将来運用: 先生または権限を持つ担当者が、審査結果に基づいて確定登録。
- 審査合格登録から、級・段更新、次の集計期間開始までを連動させる。

## 5. 追加確認事項

1. 級・段の選択肢と、一般部・少年部等での違い。
2. 級・段ごとの審査可能稽古数と、その計算起点・例外条件。
3. 同日に複数回稽古した場合の数え方。
4. 導入時の繰越稽古数を、本人申告・管理者入力のどちらにするか。
5. 審査費を既存会費データへ追加する具体的方法。
6. 集金予定額に影響する休会、途中入会、家族、複数道場等の既存ルール。
7. 合気会共通管理表について、自動生成帳票と先生の確認印で正式提出できるか。
8. 提出帳票に必要な必須項目、書式、保存期間。

## 6. 実装順序案

1. 現行コード、テーブル、ビュー、DTO、画面をRealityとして確認する。
2. 会員の級・段の保持と出席時の自己申告・更新を実装する。
3. 本日の出席一覧へ級・段、審査進捗、ソート、色分けを追加する。
4. 会員向け出席・審査進捗サマリーを追加する。
5. 会費系の既存機能を確認し、予定額と実績額の照合表示を整理する。
6. 審査費を追加する。
7. 実運用で検証後、公式管理表の自動生成へ進む。

## 7. 非機能方針

- 出席時の操作数を増やさず、変更がない場合は従来と同程度の操作で完了できること。
- 一覧表示、ソート、保存がキビキビ動くこと。
- 会員本人の情報と全会員情報の閲覧範囲を、現行の認証・権限設計に合わせて確認すること。
- 公式提出情報へ発展する段階では、修正者、修正日時、先生の確認時点、提出帳票を追跡可能にすること。
