# 1. 目的

出席、会費などの業務を直接呼び出しで結合せず、登録された事実を介して疎結合に連動させる。

本方式を「私書箱方式」と呼ぶ。

---

# 2. 基本構成

- 発行元は、業務処理の成功後に「起きた事実」をPostする。
- Post Officeは、イベントの保管と購読先への配送だけを担う。
- 関連業務は、自分の私書箱に届いたイベントを自分のタイミングで処理する。
- 業務判断や不可逆な確定は自動化せず、人が確認して実行する。

発行元は購読先を知らず、購読側も発行元の登録処理へ入り込まない。

---

# 3. Postフェーズ

業務処理は次の境界を持つ。

1. Record: 業務データを登録する。
2. Post: 登録後処理を行う。
3. PostEvent: 登録された事実をイベントとして投函する。

PostEventはイベントの投函までを責務とし、購読側の業務処理を直接実行しない。

---

# 4. 命名規約

|責務|命名|
|---|---|
|イベント投函入口|`Prefix_postEvent`|
|購読イベント処理入口|`Prefix_handleEvent`|
|購読登録入口|`Prefix_registerEventSubscriptions`|

最初の投函入口は `attendance_postEvent`、イベント種別は `ATTENDANCE_CHANGED` とする。

将来の会費候補処理は `billingCandidate_handleEvent` を入口候補とする。

---

# 5. 将来のデータ構成案

以下は名称を含めて実装時にRealityを確認し、確定する。

- イベント原簿
- イベント購読マスタ
- 購読先ごとのイベント私書箱
- 購読先ごとの処理状態、再試行回数、エラー情報

イベント全体に一つの処理済状態を持たせず、購読先ごとに状態を管理する。

---

# 6. 実行タイミング

私書箱は受動的な保管場所とする。処理の起動方法は、次から業務ごとに選択できる設計を目指す。

- 画面表示時または利用者の更新操作
- 定時実行
- 登録完了後の起動通知
- 管理者による手動実行

---

# 7. 現在のReality

- `attendance_postEvent` の入口だけを設置する。
- 現時点の入口は外部書込み、配送、購読処理を行わない。
- 出席登録結果には `posted: false` と `deferred: true` を明示する。
- イベント基盤と会費候補連動はTASK-FWK-020として保留する。
